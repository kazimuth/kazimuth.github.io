Visualizing Functions and Relations

Introduction

The universe is not veiled; all its activities lie open.

  — 圓悟克勤

Many people seem to think that math is boring, unpleasant, and difficult to understand. These people are right; but I wish they weren’t.

It’s important to understand that math is basically just a pen-and-paper board game, like dungeons and dragons. The rules of math are carefully designed to simulate the real world. It’s not magic.

But math isn’t taught this way. It’s taught as an strict and arcane collection of rules that have to be memorized, on pain of bad test scores. Not much time is spent helping students understand why we choose these rules. Various people have made the case that this is not a good way to operate ([1] [2] [3] [4] [5]). There’s a growing wave of math resources focusing on intuitive, sensory engagement with math.

I think resources like this are especially needed for modern pure mathematics. For about a century, the style in professional math has been:

  • no pictures in publications (you’re allowed to draw them for your friends cite but publishing them isn’t allowed)
  • all notation as concise and abstract as possible
  • use as many fancy tricks to describe your problem as you can think of
  • leave as much unsaid as possible

This makes pure math seem much scarier than it really is.

This post is about two important pure-mathematical tools: functions and relations. These are extremely important pieces in the board game of modern math. We’ll see different ways to understand and visualize them, go through some of the properties they can have, and find out the special moves that these properties unlock.

We’ll be mixing pictures, classic math notation, and modern formal diagrams. These are pictures that act as board game pieces, with rules for how you’re allowed to combine and modify them.

The goal of this post is to understand the following sentence:

A function is an input-total relation that preserves equality.

By “understand”, I mean both to get an intuitive understanding of what it means, and to know how to prove it – how to use it as a strategy in the game of formal math, if a pro mathematician challenges you to a duel.

Let’s begin.

What’s a relation?

First, we need to define our playing pieces. The most basic piece we’ll use is called a binary relation. Before we start writing out equations and strict mathematical rules, let’s try to get a general sense of what a binary relation looks like. There are actually multiple ways to draw them, but they’re all connected.

Describing relations

So far, what we’ve been doing doesn’t look much like traditional math. It’s all pictures! Isn’t math supposed to have equations and stuff?

The answer is: actually, it’s fine for math to be made of pictures. The original Greek mathematicians drew lots of pictures! But sometime around the late 1800s, pictures started to be considered informal. Only math made of ordered sequences of symbols counted as “real math”. There’s a modern movement to get rid of this requirement though, called diagrammatic mathematics. I love diagrammatic mathematics.

But it’s still useful to be able to make math out of symbols in a line. For one thing, it lets you explain things to mathematicians that prefer things that way. Also, computer memory is basically a sequence of boxes with symbols in them, and it’s useful to learn how to squish things so that they fit in that shape.

To write a relation in a line, we can write it as a sequence of ordered pairs (a, b) between brackets:

R = {(1, "a"), (2, "b"), (2, "c"), (3, "c")}

Alternatively, we can write it as a collection of labeled pairs (x: a, y: b):

R = {(x: 1, y: "a"), (x: 2, y: "b"), (x: 2, y: "c"), (x: 3, y: "c")}

Labeled pairs take up more space, which is probably why they’re rarely used in traditional mathematics texts. They’re common in the database literature though.

To describe our relation “R” using ordered symbols, we can write something like the following:

R is a labeled binary relation on the types 1..3 and “a”..“c” such that:

  • (x: 1, y: “a”) is in R
  • (x: 2, y: “b”) is in R
  • (x: 2, y: “c”) is in R
  • (x: 3, y: “c”) is in R
  • no other records are in R

Note: a “type” is a mathematical description of the forms a variable can take, and a “record” is the mathematical equivalent of a form with filled-in boxes:

[record, form]

We can write this more compactly:

“R: Set[(x: 1..3, y: “a”..“c”)] = { (1, “a”), (2, “b”), (2, “c”), (3, “c”) }”

A “Set” is a collection of objects in no particular order. A Set[(x: )]

For programmers, we can translate this fairly literally to the programming language Python. If you’re not a programmer, ignore the following.

from dataclasses import dataclass
from typing import Union, Literal, Set

OneToThree = Union[Literal[1], Literal[2], Literal[3]]
AToC = Union[Literal["a"], Literal["b"], Literal["c"]]

@dataclass(frozen=True)
class XY:
    x: OneToThree
    y: AToC

R: Set[XY] = {
    XY(x=1, y="a"),
    XY(x=2, y="b"),
    XY(x=2, y="c"),
    XY(x=3, y="c"),
}
print(R)

And then if we run this script, the computer will politely print out a one-line representation of R.

Defining relations by rules

You may remember equations like this from middle school:

$$y = f(x)$$

Often accompanied by graphs like this:

[graph1]

In that equation, $f$ is called a function, and $=$ is called a relation.

Functions are often explained as a type of machine that eats inputs and produces outputs:

[machine1]

We imagine the lines in this diagram as wires along which numbers can flow. The idea is, if you put the number 3, say, into the input wire, the machine will churn for a bit, and then spit out another number on the output wire, like 6.

[machine1_churning]

We give this number a new name, $f(3)$, and write $f(3) = 7$ to indicate that the machine $f$ always outputs $7$ when you feed it $3$.

It’ll be helpful for our purposes to focus on functions that go between discrete collections of objects, like whole numbers, rather than continuous ranges of objects, like the number line:

[number_plot1]

We can also use other things besides numbers, like symbols or strings of text:

[emoji_graph1]

Of course, once we start doing this, there’s no reason we have to draw functions in the classic way. We could also draw them as 2-column tables, like this:

[number_emoji_plot1]

Or as a kind of network diagram:

[number_emoji_digraph1]

Do you see the connection between these diagrams? How might we convert from one form to another?

We actually have some similar rules of equality for our other types of diagrams. For instance, I want our tables to be unordered: only the presence or absence of a row should matter, not where it is in the table. So we can write equals signs between tables with the same rows in different orders:

To recap: so far, we have five different ways to draw functions:

[equations_machines_plots_tables_digraphs]

That’s quite enough to deal with, so we’ll stop there. But you might wonder: which way is the real way to draw functions? What are functions really? Unfortunately, the answer depends on who you ask.

  • An ancient Greek mathematician might say: none of these pictures are real. Functions are actually Platonic Ideals that exist eternally outside time and space. These diagrams are all pale imitations of them – their imperfect reflections on the wavering surface of our impure world.

  • A mathematician from the 1900s might say: a function is a kind of formal term (game piece) within the formal system (game ruleset) called Zermelo-Frankel Set Theory. All of these pictures are just playful illustrations, which don’t have much to do with mathematics proper.

  • A modern, diagrammatic mathematician might say: all of these pictures of functions are real! They all contain the same information, and we can convert back and forth between them as needed, so any one of them is as good as any of the others. In fact, being able to move between them is useful for solving real problems.

A modern mathematician might draw the following diagram:

[function_diagrams_isomorphisms]

The half-squiggly equals sign $\cong$ means “contains the same information as” or “can be converted back and forth in a reversible way”. $A \cong B$ is read aloud as “A is isomorphic to B” – isomorphic translates to something like “same-shape”.

To be more explicit, we can describe strategies for converting back and forth between these different forms:

  • To go from a 2-d plot to a table, go through each marked cell, find its x and y coordinates, and write them down in a table. [plot2table]

  • To go from a table to a graph, go through each row in the table, and fill in the box with those coordinates. [table2plot]

  • To go from a 2d plot to a digraph, draw circles around the input and output labels. Go through each marked cell, and draw a bent arrow from the input label to the output label. Then, rearrange the input and output nodes and unbend the arrows. [plot2digraph]

  • To go from a digraph to a 2d plot, go through each arrow, and fill in the box corresponding to its input and output coordinates. [digraph2plot]

We can find similar recipes for moving between our other forms – equations and machine-diagrams – but we’ll need some more tools first. Also, right now we’re being informal – playing a fast and loose mathematical game.

To be more formal, we’ll need to make the rules a little stricter. We’ll design a game where diagrams are game pieces, and our conversion strategies are moves. Then we’ll be able to show that, according of the rules of the game, our conversion strategies are reversible – each of them is a move that has another move that perfectly undoes it.

This kind of thing – converting loose real-world reasoning to strict role-playing games – is called formalization or mathematical rigor. Before we formalize, though, we need to cover the other kind of piece we want to play with. It’s my favorite.

Relations

Relations are not usually explained very well. People mostly just learn the rules for $=$, and sometimes $<$, and don’t learn that these are part of a broader class of objects.

Just like functions, there are many ways to understand and picture relations. To start, I’ll just show you some pictures, so you can get an idea. Here are some pictures of relations:

[some_relations_matrix]

You may notice that these look a lot like functions: they fit into all the same kinds of pictures. So what’s the difference?

The difference is that, where a function has to take any input to exactly one output, a relation can take an input to any number of outputs: many, or zero. This is clear in the digraph picture:

[relations_vs_functions_digraph]

A similar distinction holds for the table story. A function-table has each possible input appear in exactly one row:

[relations_vs_functions_table]
(annotation: each value appears once; this value appears multiple times; this value does not appear.)

But as you can see, this isn’t true for relation-tables.

In function-plots, every column (input-value) has exactly one cell filled, but this isn’t true for relation-plots.

[relations_vs_functions_plot]

It’s slightly harder to translate relations to the traditional story of input-output machines. A relation can take one input to many outputs: does that mean the machine produces many outputs at once?

[relation_machine_spewing]

Or that the machine randomly selects an output?

[relation_machine_rolling_dice]

In my opinion, neither of these stories is quite right. They are both useful – the first one has to do with set-valued functions, and the second one has to do with probability. But I think there’s a better place to start. In my opinion, it’s best to think of relations as being constraints. Let me give an example.

Say we have a ball bearing trapped in a groove on a piece of metal:

[ball_bearing_1]

The ball can’t get out of the groove, it’s trapped, but it can move freely within the groove. We can divide the surface of the metal up, using a grid:

[ball_bearing_grid]

Now we can describe the position of the ball using grid coordinates. Constraints arise when we ask:

"If I know the horizontal position of the ball, what do I know about the vertical position of the ball?"

Depending on the shape of the groove, we may have more or less information. For instance, if we have a groove that goes in a diagonal line:

[diagonal groove]

Then when we measure the horizontal grid cell of the ball, there is only one vertical position it can be in. We know “exactly” where the ball is.

(Of course the ball can still move around inside its grid cell:

[groove_coarse]

But we’re ignoring that for now. If we wanted to keep track of that information, we’d need a finer grid.)

That shape of groove is still compatible with the idea of functions – we can think of the horizontal position as being an “input”, which uniquely determines a single “output”. But for many groove shapes this story falls apart. (In fact, most of them – see Appendix: Counting.)

For instance, if the groove has a vertical slice at one end:

[groove_function_with_vertical]

Now the horizontal position doesn’t always determine the vertical position anymore. Most of the time, it does. But if the ball is all the way to the right, then the horizontal position only gives us partial information.

For other grooves, like a circle:

[groove_circle]

The horizontal position never determines the vertical position of the ball. There’s always two options, depending on whether the ball is in the upper half or lower half of the circle.

Why is it useful to be able to think about systems like this? It turns out that systems like this emerge very commonly in science and engineering. In the natural world, variables are not cleanly divided into inputs and outputs. We may be able to measure a system in different ways, and some of these measurements can give us partial information about the other ones, but they usually don’t give us complete information. Some nice articles talking about this point of view are [TODO] – from a control systems point of view – and [TODO] – from a mathematical point of view.

So, that’s my answer to “what is a relation.” It’s just a constraint between different variables, without necessarily having inputs or outputs. So, how are we going to draw this as a machine? We can actually keep the same picture:

[relation_machine]

But now we think of it as a constraint – it allows certain combinations of values to occur on either side of it, and forbids other combinations. (If you’re familiar with electronics, this is very much like a circuit component, which imposes constraints on the voltages and currents around it, but doesn’t treat any of these as “inputs” or “outputs”.)

To write that a constraint holds between two constants, we write its name between them. For instance, 1 = 1 or 3 < 5. Or, if R is

[R_sample_table]

Then we can write [smile] R [frown]. Note: it’s bad form to write a constraint between two variables if it doesn’t actually hold. Mathematicians usually get mad at you if you write 1 = 2, for instance.

We can do a similar thing for writing that a constraint holds between variables. We can write x <= y to describe a system where the variable x must be less than or equal to y. To draw a picture of such a situation, we go through all the possible combinations of x and y, and mark the ones where the constraint holes.

[leq_rel_row_vector]

More variables

A brief side note: relations extend naturally to more than two variables.

Some of the visualizations fall apart as you add more variables though.

[too_higher_order]

We’re going to mostly focus on the two-variable case, since it’s traditional. But the higher-dimensional relations will peek through quite a bit.

Working with relations

We’ve talked a lot about what relations are, and how to visualize them, but how do we do proper math with them?

High-school math focuses on operations (+, -, *, /) and relationships (=, <) between numbers. We’re going to instead focus on operations and relationships between relations. (If you think this is meta, just wait.)

The primary skill acquired through mathematical training is the ability to fill in gaping holes in proofs. For the uninitiated, mathematics will seem to be much more hole than proof. It’s all cross-references and citations, seemingly unanchored from the world.
